补充:
一个典型的通义千问（Qwen）网页端或客户端向后端发送的对话请求参数（Request Payload）
这些字段位于最外层，用于控制整个请求的会话状态和通信方式。
chat_id: 字符串 (UUID)。当前对话窗口的唯一标识符。用于区分用户创建的不同聊天会话。
model: 字符串。指定本次对话使用的大模型版本。这里是 "qwen3.7-plus"（注：目前官方公开多为 qwen-plus/max 等，这可能是内部测试版本号或笔误）。
chat_mode: 字符串。对话模式。"normal" 表示普通对话模式，可能还有 agent（智能体）、search（搜索）等模式。
stream: 布尔值。是否开启流式输出。true 表示开启，即“打字机”效果，模型生成一个字就返回一个字，而不是等全部生成完再返回。
incremental_output: 布尔值。配合 stream: true 使用。true 表示增量返回（每次只返回新生成的文本），false 表示全量返回（每次返回从开头到当前的完整文本）。
version: 字符串。API 或通信协议的版本号，这里是 "2.1"。
parent_id: 字符串 (UUID)。顶层请求的父消息 ID。在多轮对话中，用于告诉后端当前请求是基于哪一条历史消息发起的。
timestamp: 数字。请求发起时的时间戳（秒级或毫秒级）。
二、 消息内容与结构参数 (messages 数组)
这个数组包含了当前发送的具体消息内容。现代大模型前端通常采用对话树（DAG） 结构，而不仅仅是线性的列表，因此包含了很多用于维护树结构的字段。
content: 字符串。用户实际发送的文本内容，这里是 "你可以做到什么"。
role: 字符串。消息发送者的角色。这里是 "user"（用户），还可能是 "assistant"（AI）或 "system"（系统）。
fid / id: 字符串 (UUID)。当前这条消息节点的唯一标识符（Feature ID / Forward ID）。
parentId / parent_id: 字符串 (UUID)。当前消息的父消息 ID。用于将当前消息连接到上一条消息，形成对话链条。（注：代码中同时出现了驼峰和下划线命名，可能是前端序列化时的兼容处理）。
childrenIds: 数组。当前消息的子消息 ID 列表。这是支持“对话分支”的关键。例如：你对 AI 的同一个回答不满意，点击了“重新生成”，或者你修改了历史问题，就会产生新的分支，这些分支的 ID 都会记录在父消息的 childrenIds 中。
chat_type / sub_chat_type: 字符串。聊天类型。"t2t" 代表 Text-to-Text（文本到文本），用于区分文生图、文生视频等其他模态的对话。
files: 数组。用户上传的附件（如图片、PDF文档等），这里为空 [] 表示纯文本对话。
models: 数组。该消息指定使用的模型列表。有时用于多模型对比或特定的路由分发。
user_action: 字符串。用户的动作类型。"chat" 表示正常发送消息，可能还有 "retry"（重新生成）、"edit"（编辑历史消息）等。
timestamp: 数字。该条消息创建/发送的时间戳。
三、 扩展元数据 (extra)
用于传递一些非核心的、扩展的上下文信息。
extra.meta.subChatType: 进一步细化聊天类型，这里也是 "t2t"，与外层的 sub_chat_type 呼应。
四、 高级特性控制 (feature_config)
这是控制大模型高级能力（如深度思考、联网搜索）的核心配置。
thinking_enabled: 布尔值。是否开启 “思考/推理”模式（类似 QwQ 或 DeepSeek-R1）。开启后，模型会在回答前展示内部的思考过程（CoT）。
auto_thinking: 布尔值。是否自动触发思考模式。如果为 true，模型会根据问题的复杂度自动判断是否需要进入深度思考。
output_schema: 字符串。控制输出的结构或阶段。"phase" 可能意味着分阶段输出（例如：先输出思考阶段，再输出最终回答阶段）。
research_mode: 字符串。研究/联网搜索模式。"normal" 表示普通模式，可能还有 "deep"（深度搜索/研究）等，用于控制模型是否调用搜索引擎以及搜索的深度。
