# 电商智能助手平台 - 项目概览

## 📋 项目简介

### 项目名称
**电商智能助手平台（E-commerce AI Assistant Platform）**

### 项目定位
基于多模态大语言模型的电商智能化服务平台，为电商业务提供智能客服、商品文案生成、图片审核、评论分析等 AI 能力，通过微服务架构实现高可用、可扩展的 AI 中台解决方案。

### 核心价值
- **降本增效**：自动化处理 50%+ 的客服咨询，降低人工成本
- **体验提升**：7×24 小时智能响应，用户满意度提升 30%
- **数据驱动**：实时监控 AI 调用数据，优化运营策略
- **灵活扩展**：支持多模型接入，可根据场景选择最优模型

## 📦 核心功能模块

### 1. AI 网关服务（AI Gateway Service）

**职责**：统一管理 AI 模型调用，实现智能路由和负载均衡

**核心功能**：
- ✅ 多模型接入（文本、图像、多模态）
- ✅ 智能路由策略（根据问题类型选择最优模型）
- ✅ 请求限流和配额管理
- ✅ 响应缓存（Redis）
- ✅ 失败重试和熔断降级
- ✅ 调用日志记录

**技术亮点**：
- 采用**策略模式**实现模型路由，新增模型只需实现统一接口
- 基于 **Semaphore + Redis** 实现两级限流（全局并发 + 用户配额）
- 使用 **WebClient** 异步调用外部 API，提升吞吐量
- 通过 **AOP** 自动记录调用指标（耗时、Tokens、成本）

---

### 2. 智能客服系统（Customer Service System）

**职责**：提供 7×24 小时智能客服能力

**核心功能**：
- ✅ 多轮对话管理（上下文保持）
- ✅ 常见问题自动回答
- ✅ 人工客服无缝切换
- ✅ 对话历史记录和检索
- ✅ 满意度评价收集

**应用场景**：
- 商品咨询（价格、库存、规格）

- 投诉建议处理

**技术实现**：
- 会话状态存储在 **Redis**（TTL 30分钟）
- 历史对话持久化到 **MySQL**

## 💾 数据存储设计

### MySQL 核心表结构

#### 1. 用户表（user）
```sql
CREATE TABLE user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL UNIQUE,
    email VARCHAR(100),
    api_key VARCHAR(64) UNIQUE,  -- 用户专属 API Key
    daily_quota INT DEFAULT 100,  -- 每日配额
    monthly_quota INT DEFAULT 3000,  -- 每月配额
    used_today INT DEFAULT 0,
    used_this_month INT DEFAULT 0,
    status TINYINT DEFAULT 1,  -- 1:正常 0:禁用
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_api_key (api_key)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 2. 会话记录表（conversation）
```sql
CREATE TABLE conversation (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    session_id VARCHAR(64) NOT NULL,  -- 会话 ID
    model_type VARCHAR(20) NOT NULL,  -- text/image/multimodal
    model_name VARCHAR(50),  -- 具体模型名称
    request_content TEXT,  -- 请求内容
    response_content LONGTEXT,  -- 响应内容
    image_urls JSON,  -- 图片 URL 数组
    tokens_used INT,  -- Token 消耗
    cost DECIMAL(10,4),  -- 费用（元）
    response_time_ms INT,  -- 响应时间（毫秒）
    cache_hit TINYINT DEFAULT 0,  -- 是否命中缓存
    status TINYINT DEFAULT 1,  -- 1:成功 0:失败
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_session (user_id, session_id),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 3. 调用统计表（usage_stats）
```sql
CREATE TABLE usage_stats (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    stat_date DATE NOT NULL,  -- 统计日期
    total_calls INT DEFAULT 0,  -- 总调用次数
    success_calls INT DEFAULT 0,  -- 成功次数
    failed_calls INT DEFAULT 0,  -- 失败次数
    total_tokens INT DEFAULT 0,  -- 总 Token 数
    total_cost DECIMAL(10,4) DEFAULT 0,  -- 总费用
    avg_response_time_ms INT,  -- 平均响应时间
    cache_hit_count INT DEFAULT 0,  -- 缓存命中次数
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_date (user_id, stat_date)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 4. 电商场景业务表（示例：商品文案）

CREATE TABLE product_copywriting (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,

## 🔄 消息队列设计（RabbitMQ）

### Exchange 和 Queue 规划

#### 1. 异步任务交换机（ai.task.exchange）
**类型**：Direct Exchange

| Queue | Routing Key | 用途 | 消费者 |
|-------|------------|------|--------|
| `ai.task.copywriting` | `task.copywriting` | 文案生成任务 | CopywritingConsumer |
| `ai.task.image-audit` | `task.image-audit` | 图片审核任务 | ImageAuditConsumer |
| `ai.task.review-analysis` | `task.review-analysis` | 评论分析任务 | ReviewAnalysisConsumer |

#### 2. 事件通知交换机（ai.event.exchange）
**类型**：Fanout Exchange

| Queue | 用途 | 消费者 |
|-------|------|--------|
| `ai.event.log` | 日志记录 | LogConsumer → LogStash |
| `ai.event.notification` | 用户通知 | NotificationConsumer |
| `ai.event.alert` | 告警通知 | AlertConsumer |

#### 3. 死信队列（ai.dlx.exchange）
**用途**：处理失败的任务（重试后仍失败）

| Queue | 用途 | 处理方式 |
|-------|------|---------|
| `ai.task.dlx` | 失败任务归档 | 人工介入或定时重处理 |

### 消息可靠性保证

1. **生产者确认**：开启 `publisher-confirm` 和 `publisher-return`
2. **消息持久化**：Queue 和 Message 都设置为持久化
3. **消费者 ack**：手动 ack，处理成功后才确认
4. **重试机制**：失败消息重新入队，最多重试 3 次
5. **死信队列**：超过重试次数的消息转入死信队列

## 🔐 安全设计

### 1. 认证授权
- **JWT Token**：用户登录后颁发 JWT，有效期 24 小时
- **API Key**：每个用户分配唯一 API Key，用于服务端调用
- **Spring Security**：统一权限控制，基于角色的访问控制（RBAC）

### 2. 数据安全
- **敏感信息加密**：API Key、用户密码加密存储
- **HTTPS**：全站 HTTPS 传输
- **SQL 防注入**：MyBatis 参数化查询
- **XSS 防护**：输入内容过滤和转义

### 3. 访问控制
- **IP 白名单**：管理后台限制 IP 访问
- **频率限制**：基于 Redis 的滑动窗口限流
- **配额管理**：用户级别调用次数限制
- **黑名单机制**：异常用户自动封禁

## 📈 性能优化策略

### 1. 缓存优化
- **多级缓存**：本地缓存（Caffeine）+ Redis
- **缓存预热**：系统启动时预加载常见问题
- **缓存更新**：采用 Cache-Aside 模式
- **热点 Key 保护**：防止缓存击穿

### 2. 数据库优化
- **索引优化**：为高频查询字段添加索引
- **读写分离**：主库写，从库读（可选）
- **分库分表**：数据量大时按用户 ID 分片（可选）
- **慢查询监控**：定期分析慢查询日志

### 3. 异步处理
- **非核心业务异步化**：日志记录、统计聚合、通知发送
- **批量处理**：评论分析等耗时任务批量调用 API
- **消息队列削峰**：突发流量通过 RabbitMQ 缓冲

### 4. API 调用优化
- **连接池**：WebClient 配置合理的连接池大小
- **超时控制**：设置合理的 timeout（30s）
- **重试机制**：网络波动时自动重试（最多 3 次）
- **请求合并**：相同问题合并为一个 API 调用

### 5. 前端优化
- **流式输出**：SSE 实现打字机效果，提升用户体验
- **懒加载**：图片和组件按需加载
- **CDN 加速**：静态资源走 CDN
